# Spec Shape

The resulting `SPEC.md` should be short but operational.

## Minimum Sections

- objective
- target user
- problem statement
- in-scope capabilities
- explicit non-goals
- inputs and outputs
- constraints
- acceptance criteria
- risks and open questions

## First-Phase Rule

Phase 1 should be small enough that a single Codex task tree can execute it without turning the spec into a research program.

Prefer:

- one user persona
- one primary workflow
- one or two core outputs
- a narrow validation strategy

Reject phase 1 scopes that already require a platform.
